# 产品研发流程 SOP

标准的产品研发流程，从需求到发布的完整指南。

## 流程概览

```
需求分析 → 技术设计 → 开发实现 → 测试验证 → 发布上线
   ↓           ↓           ↓           ↓           ↓
  PRD       技术方案      代码        测试报告     发布记录
```

## 阶段 1: 需求分析

### 触发条件

- 产品提出新需求
- 用户说"开始新需求"
- 项目初始化

### 执行步骤

1. **创建 PRD 文档**
   ```
   用户: "创建订单模块的 PRD"

   AI 操作:
   - 使用 doc-workflow Skill 创建 PRD
   - 应用 templates/prd.md 模版
   - 引导用户填写需求内容
   ```

2. **完善 PRD 内容**

   确保包含以下部分：
   - [ ] 背景与目标
   - [ ] 用户故事
   - [ ] 功能需求
   - [ ] 非功能需求
   - [ ] 验收标准

3. **提交评审**
   ```
   用户: "PRD 完成，提交评审"

   AI 操作:
   - 更新文档状态为 review
   - 提醒配置评审人
   - 可选：同步到飞书
   ```

4. **评审通过**
   ```
   用户: "PRD 评审通过"

   AI 操作:
   - 更新文档状态为 approved
   - 建议下一步：创建技术方案
   ```

### 交付物

- PRD 文档 (`docs/product/prd-{module}.md`)
- 状态: `approved`

### 退出条件

PRD 评审通过，进入技术设计阶段

---

## 阶段 2: 技术设计

### 触发条件

- PRD 状态变为 `approved`
- 用户说"创建技术方案"

### 执行步骤

1. **创建技术方案**
   ```
   用户: "创建订单模块的技术方案"

   AI 操作:
   - 关联到 PRD 文档
   - 应用 templates/tech-design.md 模版
   - 根据 PRD 生成初步设计
   ```

2. **完善技术方案**

   确保包含以下部分：
   - [ ] 系统架构
   - [ ] 数据模型
   - [ ] 接口设计
   - [ ] 技术选型
   - [ ] 风险评估

3. **提交评审**
   ```
   用户: "技术方案完成，提交评审"

   AI 操作:
   - 更新文档状态为 review
   - 可选：安排技术评审会议
   ```

4. **评审通过**
   ```
   用户: "技术方案评审通过"

   AI 操作:
   - 更新文档状态为 approved
   - 建议下一步：拆解开发任务
   ```

### 交付物

- 技术方案 (`docs/tech/tech-{module}.md`)
- 状态: `approved`

### 退出条件

技术方案评审通过，进入开发阶段

---

## 阶段 3: 开发实现

### 触发条件

- 技术方案状态变为 `approved`
- 用户说"开始开发"

### 执行步骤

1. **拆解开发任务**
   ```
   用户: "拆解开发任务"

   AI 操作:
   - 分析技术方案
   - 创建任务列表
   - 可选：同步到飞书 Bitable

   建议任务粒度：
   - 每个任务 1-3 天
   - 任务可独立测试
   - 明确验收标准
   ```

2. **编码实现**

   遵循开发规范：
   - 分支命名: `feature/{module}-{feature}`
   - 提交信息: 遵循 Conventional Commits
   - 代码评审: PR 必须经过评审

3. **单元测试**

   每个功能模块需要：
   - [ ] 单元测试覆盖率 > 80%
   - [ ] 关键路径有集成测试
   - [ ] 测试通过才能合并

4. **代码评审**
   ```
   用户: "提交代码评审"

   AI 操作:
   - 检查代码规范
   - 检查测试覆盖
   - 生成 PR 描述
   ```

### 交付物

- 功能代码
- 单元测试
- 代码评审记录

### 退出条件

所有开发任务完成，代码合并到主分支

---

## 阶段 4: 测试验证

### 触发条件

- 开发任务完成
- 用户说"开始测试"

### 执行步骤

1. **创建测试方案**
   ```
   用户: "创建测试方案"

   AI 操作:
   - 应用 templates/test-plan.md 模版
   - 根据 PRD 和技术方案生成用例
   ```

2. **执行测试**

   测试类型：
   - [ ] 功能测试
   - [ ] 回归测试
   - [ ] 性能测试（如需要）
   - [ ] 安全测试（如需要）

3. **记录缺陷**
   ```
   用户: "发现一个 bug"

   AI 操作:
   - 创建缺陷记录
   - 关联到相关任务
   - 同步到飞书 Bitable
   ```

4. **回归测试**

   缺陷修复后：
   - [ ] 验证缺陷已修复
   - [ ] 执行相关回归用例
   - [ ] 更新测试报告

5. **测试报告**
   ```
   用户: "生成测试报告"

   AI 操作:
   - 汇总测试结果
   - 统计缺陷情况
   - 给出发布建议
   ```

### 交付物

- 测试方案 (`docs/test/test-{module}.md`)
- 测试报告
- 缺陷记录

### 退出条件

- 所有 P0/P1 缺陷已修复
- 测试通过率 > 95%

---

## 阶段 5: 发布上线

### 触发条件

- 测试验证通过
- 用户说"准备发布"

### 执行步骤

1. **发布准备**

   检查清单：
   - [ ] 所有测试通过
   - [ ] 文档已更新
   - [ ] 发布说明已编写
   - [ ] 回滚方案已准备

2. **发布执行**
   ```
   用户: "开始发布"

   AI 操作:
   - 记录发布时间
   - 生成发布记录
   - 更新文档状态为 released
   ```

3. **发布验证**

   发布后验证：
   - [ ] 核心功能可用
   - [ ] 监控指标正常
   - [ ] 无严重告警

4. **发布记录**
   ```
   # 发布记录

   - 版本: v1.2.0
   - 时间: 2026-03-09 16:00
   - 内容:
     - 新增订单模块
     - 优化支付流程
   - 相关文档:
     - PRD: docs/product/prd-order.md
     - 技术方案: docs/tech/tech-order.md
   ```

### 交付物

- 发布记录
- 更新后的文档（状态: released）

### 退出条件

发布成功，验证通过

## 异常处理

### 需求变更

```
用户: "需求有变更"

AI 操作:
1. 记录变更内容
2. 评估影响范围
3. 更新相关文档
4. 通知相关人员
5. 必要时回退到对应阶段
```

### 进度延期

```
用户: "项目延期了"

AI 操作:
1. 分析延期原因
2. 更新任务截止日期
3. 评估对整体进度的影响
4. 建议调整方案
```

### 质量问题

```
用户: "测试发现很多 bug"

AI 操作:
1. 统计缺陷分布
2. 分析根本原因
3. 建议改进措施
4. 调整发布计划
```
