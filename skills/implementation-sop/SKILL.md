# 实施交付流程 SOP

标准的实施交付流程，从需求到交付的完整指南。

## 流程概览

```
需求调研 → 方案设计 → 实施部署 → UAT验收 → 项目交付
    ↓           ↓           ↓           ↓           ↓
 需求文档     解决方案    实施文档    测试报告    交付包
```

## 阶段 1: 需求调研

### 触发条件

- 实施团队开始新项目
- 用户说"开始新项目"
- 项目初始化，项目类型为 implementation

### 执行步骤

1. **创建需求文档**
   ```
   用户: "创建客户 ERP 实施项目"

   AI 操作:
   - 使用 doc-workflow Skill 创建需求文档
   - 应用 templates/requirement.md 模版
   - 引导用户填写客户背景和需求
   ```

2. **完善需求内容**

   确保包含以下部分：
   - [ ] 客户背景
   - [ ] 业务现状
   - [ ] 痛点分析
   - [ ] 需求清单
   - [ ] 优先级

3. **提交内部评审**
   ```
   用户: "需求完成，提交评审"

   AI 操作:
   - 更新文档状态为 review
   - 提醒配置评审人
   - 可选：同步到飞书
   ```

4. **客户确认**
   ```
   用户: "客户已确认需求"

   AI 操作:
   - 更新文档状态为 approved
   - 建议下一步：创建解决方案
   ```

### 交付物

- 需求文档 (`docs/requirement/requirement-{customer}.md`)
- 状态: `approved`

### 退出条件

需求文档获得客户确认，进入方案设计阶段

---

## 阶段 2: 方案设计

### 触发条件

- 需求文档已确认
- 用户说"开始设计方案"

### 执行步骤

1. **创建解决方案文档**
   ```
   用户: "创建 ERP 解决方案"

   AI 操作:
   - 使用 doc-workflow Skill 创建解决方案
   - 应用 templates/solution.md 模版
   - 引导用户填写方案内容
   ```

2. **完善方案内容**

   确保包含以下部分：
   - [ ] 整体架构
   - [ ] 技术选型
   - [ ] 实施计划
   - [ ] 风险分析
   - [ ] 资源估算

3. **提交客户评审**
   ```
   用户: "方案完成，提交客户评审"

   AI 操作:
   - 更新文档状态为 review
   - 准备客户汇报材料
   ```

4. **客户批准**
   ```
   用户: "客户已批准方案"

   AI 操作:
   - 更新文档状态为 approved
   - 建议下一步：开始实施
   ```

### 交付物

- 解决方案文档 (`docs/solution/solution-{project}.md`)
- 状态: `approved`

### 退出条件

解决方案文档获得客户批准，进入实施部署阶段

---

## 阶段 3: 实施部署

### 触发条件

- 解决方案已批准
- 用户说"开始实施"

### 执行步骤

1. **创建实施文档**
   ```
   用户: "开始实施"

   AI 操作:
   - 使用 doc-workflow Skill 创建实施文档
   - 应用 templates/config.md 或 templates/deployment.md 模版
   - 记录实施步骤和配置
   ```

2. **实施执行**

   记录关键步骤：
   - [ ] 环境准备
   - [ ] 系统配置
   - [ ] 数据迁移（如有）
   - [ ] 功能测试
   - [ ] 性能测试

3. **提交 UAT**
   ```
   用户: "实施完成，提交 UAT"

   AI 操作:
   - 更新文档状态为 review
   - 准备 UAT 测试用例
   ```

### 交付物

- 实施文档 (`docs/implementation/implementation-{project}.md`)
- 配置清单 (`docs/config/config-{module}.md`)

### 退出条件

UAT 测试通过，进入验收阶段

---

## 阶段 4: UAT验收

### 触发条件

- 实施完成，提交 UAT

### 执行步骤

1. **创建 UAT 测试报告**
   ```
   用户: "UAT 完成"

   AI 操作:
   - 使用 doc-workflow Skill 创建测试报告
   - 应用 templates/test-report.md 模版
   - 记录测试结果
   ```

2. **测试结果记录**

   确保包含：
   - [ ] 测试用例执行情况
   - [ ] 发现的问题
   - [ ] 问题解决状态
   - [ ] 测试结论

3. **客户验收签字**
   ```
   用户: "客户 UAT 通过"

   AI 操作:
   - 更新测试报告状态为 approved
   - 更新实施文档状态为 released
   ```

### 交付物

- UAT 测试报告 (`docs/test-report/test-report-{project}.md`)
- 状态: `approved`

### 退出条件

客户 UAT 通过，项目进入交付阶段

---

## 阶段 5: 项目交付

### 触发条件

- UAT 验收通过

### 执行步骤

1. **创建交付文档**
   ```
   用户: "开始项目交付"

   AI 操作:
   - 使用 doc-workflow Skill 创建交付文档
   - 应用 templates/handover.md 模版
   - 整理交付内容
   ```

2. **交付内容清单**

   确保包含：
   - [ ] 交付物清单
   - [ ] 系统账号
   - [ ] 运维手册
   - [ ] 培训材料（如有）
   - [ ] 验收单

3. **完成交付**
   ```
   用户: "项目交付完成"

   AI 操作:
   - 更新交付文档状态为 released
   - 更新项目状态为 completed
   - 建议：创建复盘文档（可选）
   ```

### 交付物

- 交付文档 (`docs/handover/handover-{project}.md`)
- 状态: `released`

---

## 状态流转图

```
需求调研 → 方案设计 → 实施部署 → UAT验收 → 项目交付
    ↓           ↓           ↓           ↓           ↓
  draft     draft       draft      draft       draft
    ↓           ↓           ↓           ↓           ↓
  review    review      review     review      released
    ↓           ↓           ↓           ↓
 approved   approved    approved   approved
```

## 协作模式建议

| 阶段 | 推荐模式 | 说明 |
|------|----------|------|
| 需求调研 | Mode A | 现场沟通为主，AI 记录整理 |
| 方案设计 | Mode B | 人主导，AI 辅助 |
| 实施部署 | Mode C | AI 生成文档，人确认 |
| UAT验收 | Mode D | AI 生成测试报告 |
| 项目交付 | Mode D | AI 生成交付文档 |

---

## 触发下一步

当一个阶段完成时，AI 应该主动建议下一步：

```
✅ 需求调研完成

建议下一步:
[1] 创建解决方案 (使用 doc-workflow)
[2] 查看项目整体进展 (使用 project-status)
[3] 同步到飞书
```
