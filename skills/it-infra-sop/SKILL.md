# IT 信息化项目流程 SOP

标准的IT信息化项目流程，从需求到运维的完整指南。

## 流程概览

```
需求分析 → 技术方案 → 开发实施 → 测试验收 → 上线部署 → 运维支持
    ↓           ↓           ↓           ↓           ↓           ↓
 需求文档     技术方案    实施文档    测试报告    部署文档    运维手册
```

## 阶段 1: 需求分析

### 触发条件

- 开始新的IT信息化项目
- 用户说"开始新IT项目"
- 项目初始化，项目类型为 it-infra

### 执行步骤

1. **创建需求文档**
   ```
   用户: "创建CRM系统对接项目"

   AI 操作:
   - 使用 doc-workflow Skill 创建需求文档
   - 应用 templates/requirement.md 模版
   - 引导用户说明业务背景和需求
   ```

2. **完善需求内容**

   确保包含以下部分：
   - [ ] 业务背景
   - [ ] 现状分析
   - [ ] 痛点梳理
   - [ ] 需求清单
   - [ ] 优先级定义

3. **提交内部评审**
   ```
   用户: "需求完成，提交评审"

   AI 操作:
   - 更新文档状态为 review
   - 提醒配置评审人
   - 可选：同步到飞书
   ```

4. **评审通过**
   ```
   用户: "需求评审通过"

   AI 操作:
   - 更新文档状态为 approved
   - 建议下一步：创建技术方案
   ```

### 交付物

- 需求文档 (`docs/requirement/requirement-{system}.md`)
- 状态: `approved`

### 退出条件

需求文档评审通过，进入技术方案阶段

---

## 阶段 2: 技术方案

### 触发条件

- 需求文档状态变为 `approved`
- 用户说"开始设计方案"

### 执行步骤

1. **创建技术方案**
   ```
   用户: "创建CRM对接技术方案"

   AI 操作:
   - 使用 doc-workflow Skill 创建技术方案
   - 应用 templates/tech-design.md 模版
   - 引导用户填写技术细节
   ```

2. **完善技术方案**

   确保包含以下部分：
   - [ ] 系统架构
   - [ ] 接口设计
   - [ ] 数据模型
   - [ ] 技术选型
   - [ ] 安全方案
   - [ ] 风险评估

3. **提交技术评审**
   ```
   用户: "技术方案完成，提交评审"

   AI 操作:
   - 更新文档状态为 review
   - 准备技术评审材料
   ```

4. **评审通过**
   ```
   用户: "技术方案评审通过"

   AI 操作:
   - 更新文档状态为 approved
   - 建议下一步：开始实施
   ```

### 交付物

- 技术方案 (`docs/tech/tech-{system}.md`)
- 状态: `approved`

### 退出条件

技术方案评审通过，进入开发实施阶段

---

## 阶段 3: 开发实施

### 触发条件

- 技术方案状态变为 `approved`
- 用户说"开始实施"

### 执行步骤

1. **创建实施文档**
   ```
   用户: "开始实施"

   AI 操作:
   - 使用 doc-workflow Skill 创建实施文档
   - 应用 templates/integration.md 或 templates/migration.md 模版
   - 记录实施步骤
   ```

2. **实施执行**

   记录关键步骤：
   - [ ] 环境准备
   - [ ] 系统配置
   - [ ] 接口开发
   - [ ] 数据迁移（如有）
   - [ ] 联调测试

3. **提交测试**
   ```
   用户: "实施完成，提交测试"

   AI 操作:
   - 更新文档状态为 review
   - 准备测试用例
   ```

### 交付物

- 实施文档 (`docs/implementation/implementation-{system}.md`)
- 配置清单 (`docs/config/config-{module}.md`)

### 退出条件

开发完成，进入测试验收阶段

---

## 阶段 4: 测试验收

### 触发条件

- 开发实施完成
- 用户说"开始测试"

### 执行步骤

1. **创建测试报告**
   ```
   用户: "测试完成"

   AI 操作:
   - 使用 doc-workflow Skill 创建测试报告
   - 应用 templates/test-report.md 模版
   - 记录测试结果
   ```

2. **测试结果记录**

   确保包含：
   - [ ] 测试用例执行情况
   - [ ] 发现的问题
   - [ ] 问题解决状态
   - [ ] 测试结论

3. **验收通过**
   ```
   用户: "UAT 通过"

   AI 操作:
   - 更新测试报告状态为 approved
   - 建议下一步：上线部署
   ```

### 交付物

- 测试报告 (`docs/test/test-report-{system}.md`)
- 状态: `approved`

### 退出条件

测试验收通过，进入上线部署阶段

---

## 阶段 5: 上线部署

### 触发条件

- 测试验收通过
- 用户说"准备上线"

### 执行步骤

1. **创建部署文档**
   ```
   用户: "开始部署"

   AI 操作:
   - 使用 doc-workflow Skill 创建部署文档
   - 应用 templates/deployment.md 模版
   - 记录部署步骤
   ```

2. **部署内容**

   确保包含：
   - [ ] 部署步骤
   - [ ] 配置文件
   - [ ] 回滚方案
   - [ ] 验证清单

3. **执行部署**
   ```
   用户: "开始上线"

   AI 操作:
   - 记录部署时间
   - 生成部署记录
   - 更新文档状态为 released
   ```

4. **上线验证**

   部署后验证：
   - [ ] 核心功能可用
   - [ ] 监控指标正常
   - [ ] 无告警

### 交付物

- 部署文档 (`docs/deployment/deployment-{system}.md`)
- 状态: `released`

### 退出条件

部署成功，系统上线

---

## 阶段 6: 运维支持

### 触发条件

- 系统上线
- 用户说"开始运维"

### 执行步骤

1. **创建运维文档**
   ```
   用户: "创建运维手册"

   AI 操作:
   - 使用 doc-workflow Skill 创建运维文档
   - 应用 templates/operation.md 模版
   - 记录运维内容
   ```

2. **运维内容**

   确保包含：
   - [ ] 系统概述
   - [ ] 操作指南
   - [ ] 监控指标
   - [ ] 故障处理
   - [ ] 联系方式

3. **交接确认**
   ```
   用户: "运维交接完成"

   AI 操作:
   - 更新运维文档状态为 released
   - 标记项目阶段为运维中
   ```

### 交付物

- 运维文档 (`docs/operation/operation-{system}.md`)
- 状态: `released`

---

## 状态流转图

```
需求分析 → 技术方案 → 开发实施 → 测试验收 → 上线部署 → 运维支持
    ↓           ↓           ↓           ↓           ↓           ↓
  draft      draft       draft      draft       draft       draft
    ↓           ↓           ↓           ↓           ↓           ↓
  review     review      review     review     review      released
    ↓           ↓           ↓           ↓           ↓
 approved   approved    approved   approved   released
```

## 协作模式建议

| 阶段 | 推荐模式 | 说明 |
|------|----------|------|
| 需求分析 | Mode A | 现场沟通为主，AI 记录整理 |
| 技术方案 | Mode B | 人主导，AI 辅助 |
| 开发实施 | Mode C | AI 生成文档，人确认 |
| 测试验收 | Mode D | AI 生成测试报告 |
| 上线部署 | Mode D | AI 生成部署文档 |
| 运维支持 | Mode D | AI 生成运维手册 |

---

## 触发下一步

当一个阶段完成时，AI 应该主动建议下一步：

```
✅ 需求分析完成

建议下一步:
[1] 创建技术方案 (使用 doc-workflow)
[2] 查看项目整体进展 (使用 project-status)
[3] 同步到飞书
```

## 周期性任务

IT 项目通常有周期性维护任务：

```
📅 周期性任务:

本周任务:
- [ ] 周一: 检查系统监控
- [ ] 周三: 查看告警记录
- [ ] 周五: 巡检报告

建议使用 task-management Skill 管理周期性任务。
```
